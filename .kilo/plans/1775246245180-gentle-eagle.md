### NEW ISSUES TO FIX

### 1. DMA Channels API - N+1 Query Problem (CRITICAL - causes 30+ sec load)
**File:** `rescuegrid/app/api/dma/channels/route.ts`

For each victim report, task force, and volunteer, it makes 3 separate queries:
- Last message query
- Unread count query
- Flagged status query

**Example:** 20 victims + 20 TFs + 20 volunteers = 60 main queries + 180 sub-queries = 240 total sequential queries!

### 2. DMA Messages Page - Sequential Fetching
**File:** `rescuegrid/app/(dma)/dma/messages/page.tsx` lines 173-180
```typescript
fetchMessages();      // Sequential
fetchChannelData();   // Sequential  
fetchTfMembers();     // Sequential
markAsRead();         // Sequential
```
Should be parallel with `Promise.all()`

### 3. DMA Messages Page - Inefficient Realtime
**File:** `rescuegrid/app/(dma)/dma/messages/page.tsx` lines 188-242

On INSERT/UPDATE, calls `fetchMessages()` and `fetchChannels()` which refetches EVERYTHING instead of directly appending the new message.

### 4. DMA Messages Page - Scrollable Layout Issue (CRITICAL)
**File:** `rescuegrid/app/(dma)/dma/messages/page.tsx`

**User Description:** The typing/input area moves down when messages are sent - it should be FIXED at bottom like WhatsApp. Only the messages area should scroll.

**Current Layout (BROKEN):**
```
<div className="flex-1 flex flex-col bg-void">  ← Main container
  {/* Thread Header */}
  {/* TF Members Panel */}
  {/* Messages Area - flex-1, overflow-y-auto */}
  {/* Message Input - should be FIXED at bottom */}
</div>
```

**Problem:** The outer flex container with `flex-col` doesn't properly constrain height. When messages grow, the input gets pushed down instead of staying fixed.

**Fix:** 
1. Use `height: 100dvh` on the outer container
2. Ensure `overflow: hidden` on parent
3. The input area should have `shrink-0` and NOT grow
4. Messages area should have `flex-1 overflow-y-auto` with `min-height: 0`

This is the same pattern used in WhatsApp:
- Header: fixed height (`shrink-0`)
- Messages: flexible height, scrollable (`flex-1 overflow-y-auto`)
- Input: fixed height (`shrink-0`)

### 5. DMA Messages Page - No Jump-to-Bottom
Auto-scroll always goes to bottom even if user scrolled up to read older messages.

### 5. CRITICAL BUG: Volunteer Inbox Task Force Filter Invalid
**File:** `rescuegrid/app/(volunteer)/volunteer/inbox/page.tsx`

The Supabase postgres_changes filter syntax is INVALID:
```typescript
filter: `task_force_id=eq.${taskForces.map(tf => tf.id).join('&task_force_id=eq.')}`,
```
This creates: `task_force_id=eq.tf1&task_force_id=eq.tf2` - which is NOT valid Supabase OR syntax!

Supabase `postgres_changes` filter only supports ONE condition per column. Must use separate channel per task force OR use Supabase `in()` filter differently.

### 6. CRITICAL BUG: DMA Messages to Volunteer Not Reaching Volunteer App
**Root Cause Analysis:**
- DMA sends direct message: `receiver_id = volunteer's ID`, `sender_type = "dma"`
- Volunteer inbox subscribes: `receiver_id=eq.${volunteerId}`
- This SHOULD work, but...

**Likely issues:**
1. Volunteer inbox task force realtime filter bug (above) causes the entire useEffect to fail
2. Or volunteer session cookie not being set correctly
3. Or realtime channel not properly connecting

### Phase 4: Fix Volunteer Inbox Task Force Filter

**File:** `rescuegrid/app/(volunteer)/volunteer/inbox/page.tsx`

**Problem:** Invalid Supabase filter syntax for multiple task force IDs

**Solution:** Subscribe to a single channel with filter that matches ALL task force messages, then filter client-side:
```typescript
// Subscribe to ALL task force messages (no filter - handle client-side)
supabase.channel(`volunteer-taskforces-${volunteerId}`)
  .on('postgres_changes', {
    event: 'INSERT',
    schema: 'public',
    table: 'message',
    filter: 'task_force_id=not.is.null'  // or subscribe to each TF individually
  }, (payload) => {
    // Filter client-side to only add if TF is in volunteer's TF list
    const newMsg = payload.new;
    if (newMsg.task_force_id && taskForces.find(tf => tf.id === newMsg.task_force_id)) {
      // Add message
    }
  })
```

Or better: Use a separate channel per task force (cleaner Supabase-wise):
```typescript
taskForces.forEach(tf => {
  supabase.channel(`tf-messages-${tf.id}`)
    .on('postgres_changes', {
      event: 'INSERT',
      schema: 'public',
      table: 'message',
      filter: `task_force_id=eq.${tf.id}`
    }, handler)
    .subscribe();
});
```

### DMA Messages Page - Direct Append Realtime
```typescript
.on('postgres_changes', {
  event: 'INSERT',
  table: 'message',
  filter: `victim_report_id=eq.${activeChannel.id}`
}, (payload) => {
  // Directly append new message to state (O(1))
  setMessages(prev => {
    if (prev.find(m => m.id === payload.new.id)) return prev;
    return [...prev, payload.new];
  });
  
  // Update channel in list (last_message, unread_count)
  setChannels(prev => prev.map(ch => 
    ch.id === activeChannel.id 
      ? { ...ch, last_message: payload.new.content, last_message_time: payload.new.created_at }
      : ch
  ));
});
```

### DMA Jump-to-Bottom UI
```typescript
const [isNearBottom, setIsNearBottom] = useState(true);

const handleScroll = () => {
  const { scrollTop, scrollHeight, clientHeight } = container;
  setIsNearBottom(scrollHeight - scrollTop - clientHeight < 100);

  scrollToBottom();
} else {
  setNewMessagesCount(prev => prev + 1);
```
