# Volunteer Filtering & Map Rendering Enhancement Plan

## Current State Analysis

### 1. Database Schema & Indexing

**Current Tables:**
- `volunteer`: stores lat/lng as `double precision`, skills/equipment as `text` (comma-separated)
- `assignment`: connects volunteers to tasks
- `task_force_member`: junction table for team assignments

**Current Indexes** (`002_performance_indexes.sql`):
```sql
-- Status filtering (most common query)
CREATE INDEX idx_volunteer_status_active ON public.volunteer (status) WHERE status = 'active';

-- Name text search
CREATE INDEX idx_volunteer_name_lower ON public.volunteer (lower(name));

-- Composite index for geospatial queries
CREATE INDEX idx_volunteer_location ON public.volunteer (latitude, longitude) 
  WHERE latitude IS NOT NULL AND longitude IS NOT NULL;
```

**Problems Identified:**
1. ❌ No PostGIS - uses manual Haversine formula (SQL function `calculate_distance`)
2. ❌ Skills/equipment in comma-separated text - can't use indexes (uses `ILIKE` queries)
3. ❌ Distance sorting happens in application code after fetching records
4. ❌ No partial index for location-based queries combined with status

### 2. Volunteer Search API (`/api/volunteer/search/route.ts`)

**Current Flow:**
1. Bounding box filter (latDelta/lngDelta) using `idx_volunteer_location`
2. ILIKE filters for skills/equipment (full table scan on results)
3. Fetch records with limit (default 20, max 50)
4. Post-query: Calculate Haversine distance in JS for each result
5. Sort by distance in JS
6. Filter by radius in JS

**Performance Issues:**
- Bounding box can return many records before filtering by radius
- Skills/equipment ILIKE can't use indexes efficiently
- Distance calculation happens after fetch (can't limit results early)

### 3. Assignment Creation Scoring (`CreateAssignmentModal.tsx:212-237`)

**Current Scoring (Client-side only):**
```typescript
score = Math.max(0, 30 - distance_km * 2)           // Distance
        + matches.length * 15                        // Skills
        + equipmentMatches.length * 10               // Equipment
        + (status === "active" ? 10 : 0)            // Status bonus
        + (status === "on-mission" ? -20 : 0)       // Status penalty
```

**Problems:**
- Scoring happens AFTER fetching all volunteers
- Not using the API scoring - client-side only for task force creation
- No database-level ranking

### 4. Map Rendering (`MapboxMap.tsx`)

**Current Implementation:**
- Renders ALL volunteers from `useVolunteers()` hook
- No clustering - individual markers for every volunteer
- No zoom-based layer switching
- Real-time updates via Supabase subscriptions
- `volunteersVersion` state forces re-render on any change

**Problems:**
- Performance degrades linearly with volunteer count
- No heatmap/cluster view for overview
- No top-N limiting for dense areas

## Proposed Solution (Non-Breaking Migration)

### Phase 1: Database Enhancements (Backward Compatible)

#### 1.1 New Normalized Skill System (Additive)

**New Tables (doesn't break existing):**
```sql
-- Hierarchical skill categories
CREATE TABLE skill_category (
  id text PRIMARY KEY,  -- 'medical', 'rescue', 'logistics', etc.
  name text NOT NULL,
  color text,  -- for map visualization
  priority int DEFAULT 0  -- for ranking
);

-- Individual skills with tiers
CREATE TABLE skill (
  id text PRIMARY KEY,
  category_id text REFERENCES skill_category(id),
  name text NOT NULL,
  tier int DEFAULT 2 CHECK (tier IN (1, 2, 3, 4)),  -- T1=untrained, T4=professional
  requires_certification boolean DEFAULT false
);

-- Junction table (additive - keep old skills column)
CREATE TABLE volunteer_skill (
  volunteer_id uuid REFERENCES volunteer(id),
  skill_id text REFERENCES skill(id),
  tier_override int,  -- volunteer's specific tier for this skill
  certified_at timestamp,
  expires_at timestamp,
  PRIMARY KEY (volunteer_id, skill_id)
);
```

**Migration Strategy:**
```sql
-- Migrate existing skills automatically
INSERT INTO volunteer_skill (volunteer_id, skill_id, tier_override)
SELECT 
  v.id,
  TRIM(s.skill),
  2  -- Default to T2 (basic trained)
FROM volunteer v
CROSS JOIN LATERAL unnest(string_to_array(v.skills, ',')) AS s(skill)
WHERE v.skills IS NOT NULL;
```

#### 1.2 Performance Indexes (New, Non-Breaking)

```sql
-- Covering index for location + status queries
CREATE INDEX idx_volunteer_location_status 
ON volunteer (latitude, longitude, status) 
WHERE latitude IS NOT NULL AND longitude IS NOT NULL;

-- Partial index for active volunteers with location
CREATE INDEX idx_volunteer_active_located 
ON volunteer (latitude, longitude) 
WHERE status = 'active' AND latitude IS NOT NULL AND longitude IS NOT NULL;

-- Index for last_seen (for availability scoring)
CREATE INDEX idx_volunteer_last_seen 
ON volunteer (last_seen DESC NULLS LAST) 
WHERE status = 'active';

-- Index on volunteer_skill for join queries
CREATE INDEX idx_volunteer_skill_volunteer ON volunteer_skill(volunteer_id);
CREATE INDEX idx_volunteer_skill_skill ON volunteer_skill(skill_id);
```

#### 1.3 Database-Level Scoring Function

```sql
-- Compute composite score at query time
CREATE OR REPLACE FUNCTION calculate_volunteer_score(
    vol_record volunteer,
    target_lat double precision,
    target_lng double precision,
    max_radius_km double precision DEFAULT 100,
    required_skills text[] DEFAULT '{}',
    required_equipment text[] DEFAULT '{}'
) RETURNS TABLE (
    score double precision,
    distance_km double precision,
    skill_match_count int,
    equipment_match_count int
) AS $$
DECLARE
    dist double precision;
    proximity_score double precision;
    skill_score double precision;
    equip_score double precision;
    availability_score double precision;
BEGIN
    -- Calculate distance
    dist := calculate_distance(vol_record.latitude, vol_record.longitude, target_lat, target_lng);
    
    -- Proximity score: 1.0 at center, decaying to 0 at max_radius
    proximity_score := GREATEST(0, 1 - (dist / max_radius_km));
    
    -- Skill matching (case insensitive)
    SELECT COUNT(*) INTO skill_match_count
    FROM unnest(required_skills) req
    WHERE vol_record.skills ILIKE '%' || req || '%';
    skill_score := LEAST(skill_match_count::double precision / NULLIF(array_length(required_skills, 1), 0), 1.0);
    
    -- Equipment matching
    SELECT COUNT(*) INTO equipment_match_count
    FROM unnest(required_equipment) req
    WHERE vol_record.equipment ILIKE '%' || req || '%';
    equip_score := LEAST(equipment_match_count::double precision / NULLIF(array_length(required_equipment, 1), 0), 1.0);
    
    -- Availability score based on status and last_seen
    IF vol_record.status = 'active' THEN
        IF vol_record.last_seen IS NULL OR vol_record.last_seen > NOW() - INTERVAL '5 minutes' THEN
            availability_score := 1.0;
        ELSIF vol_record.last_seen > NOW() - INTERVAL '30 minutes' THEN
            availability_score := 0.7;
        ELSE
            availability_score := 0.4;
        END IF;
    ELSIF vol_record.status = 'standby' THEN
        availability_score := 0.5;
    ELSE
        availability_score := 0.0;
    END IF;
    
    -- Composite score (user's formula: 0.40×skill + 0.35×proximity + 0.25×availability)
    score := (0.40 * skill_score * 100) + 
             (0.35 * proximity_score * 100) + 
             (0.25 * availability_score * 100);
    
    distance_km := dist;
    
    RETURN NEXT;
END;
$$ LANGUAGE plpgsql STABLE;
```

### Phase 2: API Enhancements (Backward Compatible)

#### 2.1 Enhanced Volunteer Search Endpoint

**New Query Parameters (additive):**
- `top_n`: Return only top N by score (default 50)
- `use_scoring`: Enable composite scoring (default true)
- `min_score`: Minimum score threshold (0-100)

**New Response Fields:**
```typescript
{
  data: Volunteer[],
  total: number,
  limit: number,
  offset: number,
  hasMore: boolean,
  scored: boolean,  // true if scoring was applied
  maxRadius: number, // actual radius used
  appliedFilters: {  // what filters were applied
    status: string,
    skills: string[],
    radius: number,
    scoreThreshold: number
  }
}
```

**Implementation Strategy:**
- Keep existing endpoint as fallback
- Add feature flag `USE_NEW_SCORING` (env var)
- Run both in parallel initially, compare results

#### 2.2 New Map-Optimized Endpoint

**New Endpoint:** `/api/volunteer/map/locations`

**Purpose:** Return volunteers optimized for map display with zoom-based limiting

**Query Parameters:**
- `zoom`: Current map zoom level
- `bbox`: Bounding box (minLng, minLat, maxLng, maxLat)
- `max_results`: Maximum to return (default 50)
- `cluster_threshold`: Return clusters instead of points if count > threshold

**Response:**
```typescript
{
  type: 'clusters' | 'points',
  zoom: number,
  clusters?: {
    lat: number,
    lng: number,
    count: number,
    dominant_skill: string,
    avg_score: number
  }[],
  points?: Volunteer[]
}
```

#### 3.3 Cluster Layer (Zoom 10-12)

**Using Supercluster:**
```typescript
import Supercluster from 'supercluster';

const cluster = new Supercluster({
  radius: 40,
  maxZoom: 12,
  map: (props) => ({ score: props.score, dominantSkill: props.dominantSkill }),
  reduce: (acc, props) => {
    acc.totalScore += props.score;
    acc.count += 1;
    // Track dominant skill
    if (!acc.skills[props.dominantSkill]) acc.skills[props.dominantSkill] = 0;
    acc.skills[props.dominantSkill]++;
  }
});
```

**Cluster Visualization:**
- Circle with count badge
- Color based on dominant skill category
- Size based on cluster count

### Phase 5: Migration & Rollback Strategy

#### 5.1 Database Migration Order

1. **Create new tables** (skill_category, skill, volunteer_skill) - safe, no impact
2. **Add new columns** to volunteer (optional skill_tier enum) - nullable, no impact
3. **Create new indexes** - can be done concurrently with `CREATE INDEX CONCURRENTLY`
4. **Create scoring function** - pure function, no data change
5. **Backfill volunteer_skill** from existing skills text column - batch in chunks
6. **Verify backfill** - compare counts
7. **Mark complete** - new code starts using normalized tables

#### 5.2 API Deployment Strategy

**Feature Flags:**
```typescript
const FEATURES = {
  USE_DB_SCORING: process.env.USE_DB_SCORING === 'true',
  USE_MAP_CLUSTERS: process.env.USE_MAP_CLUSTERS === 'true',
  USE_SKILL_NORMALIZATION: process.env.USE_SKILL_NORMALIZATION === 'true'
};
```

**Gradual Rollout:**
1. Deploy new code with all flags OFF (no change)
2. Enable USE_DB_SCORING for 10% of requests (A/B test)
3. Monitor: query time, result quality, error rates
4. Increase to 50%, then 100%
5. Repeat for other features

#### 5.3 Rollback Plan

**Database:**
- All changes are additive (new tables, new indexes, new function)
- To rollback: set feature flags to false
- Old code continues to use skills text column

**API:**
- Keep old scoring logic in code, just don't use it
- Switch via feature flag instantly

**Frontend:**
- New components only render if feature flag enabled
- Falls back to current MapboxMap if clustering fails
