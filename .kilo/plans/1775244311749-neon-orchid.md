# Plan: Fix DMA Messaging Panel & Victim Message Display

## Context
Victims search by phone number to find their report. No authentication needed. When they click on their report, they should see all messages for that report thread.

## Issues Identified

1. **DMA Messaging Panel shows nothing** - Channel list empty or not loading
2. **Victim sends message but cannot see it in their UI** - Message saved to DB but not visible
3. **DMA cannot see victim messages in the messaging panel** - Messages not appearing in channels
4. **DMA cannot send replies in victim threads** - Need to verify sending works

## Root Cause Analysis

### Issue 1: DMA Channels Not Loading
- `/api/dma/channels` fetches victim_reports and creates channels for each
- The API uses `service_role` client so RLS shouldn't block it
- But there may be silent failures or the channel key `${channel.type}-${channel.id}` may not match when clicked

### Issue 2: Victim Cannot See Sent Messages (PRIMARY BUG)
- Victim uses Supabase client directly (anon key) to fetch messages
- RLS policies on `message` table do NOT allow anon users to SELECT messages
- Policy `volunteer_select_message_direct` only allows SELECT where `sender_id = auth.uid()` but victim messages have `sender_id = null`
- Policy `volunteer_select_message_taskforce` requires `task_force_id` which victim messages don't have
- **Result: Victim CAN insert messages but CANNOT read them back due to RLS**

### Issue 3: DMA Cannot See Victim Messages
- The channels API fetches victim_reports correctly
- The realtime subscription in DMA messages page listens for all message INSERTs
- Most critical: victim messages have `sender_id = null`, which causes no sender name to display
- The `/api/dma/message` GET endpoint doesn't JOIN to get sender information

### Issue 4: Seed Data Bug
- Line 44 in seed.sql: DMA reply has `receiver_id = 'c1111111-...' (victim_report UUID)` instead of `NULL`
- This is wrong - DMA→victim replies should have `receiver_id = NULL`

### Issue 5: DMA Message API Missing Sender Names
- The volunteer chat correctly JOINs to get `sender:volunteer(id, name, type)`
- The DMA message API (`/api/dma/message`) does NOT include this JOIN
- Result: victim messages show "Victim" as fallback, DMA messages show nothing as sender_name

## Fix Plan

### Step 1: Fix RLS Policies for Victim Message Access (CRITICAL)
**File:** `supabase/migrations/002_rls_policies.sql`

Add policy allowing anon to SELECT messages for their own report thread:
```sql
-- Victim: SELECT messages for their own report thread (by victim_report_id)
CREATE POLICY "anon_select_message_victim_thread" ON message
  FOR SELECT TO anon
  USING (victim_report_id IS NOT NULL);
```

### Step 2: Fix DMA Channels API with Better Debugging
**File:** `rescuegrid/app/api/dma/channels/route.ts`

- Add error logging to debug failures
- Ensure it returns all victim_reports even if they have no messages
- Return channels sorted by most recent activity
- Verify victim_report rows are being fetched properly

### Step 3: Fix Sender Name Resolution in DMA Message API
**File:** `rescuegrid/app/api/dma/message/route.ts` (GET)

For victim_thread messages:
- JOIN with `victim_report` to get `phone_no` as the sender identifier
- Return `sender_name` as the phone number for victim messages

For taskforce_room messages:
- JOIN with `volunteer` to get `name` as the sender identifier (like volunteer chat does)

For direct messages:
- JOIN with `volunteer` to get `name`

### Step 4: Fix DMA Message Page Realtime Subscription
**File:** `rescuegrid/app/(dma)/dma/messages/page.tsx`

- When active channel is victim_thread, subscribe with filter: `victim_report_id=eq.${activeChannel.id}`
- When active channel is taskforce_room, subscribe with filter: `task_force_id=eq.${activeChannel.id}`
- When active channel is direct, subscribe with filter: `receiver_id=eq.${activeChannel.id}` or `sender_id=eq.${activeChannel.id}`
- Update `read_at` when DMA opens a thread (send PATCH to mark messages as read)
- Display sender identifier (phone number from victim_report) for victim messages

### Step 5: Add Mark as Read Functionality
**File:** `rescuegrid/app/api/dma/message/read/route.ts` (new API route)

- Accept channel_type and channel_id
- Mark all messages in that channel as read (set read_at = now())
- Called when DMA opens a thread

### Step 6: Fix Victim Status Page Message Display
**File:** `rescuegrid/app/(victim)/report/status/[id]/page.tsx`

- Verify realtime subscription is correctly filtering: `victim_report_id=eq.${reportId}`
- Ensure message INSERT events are being received
- No RLS change needed once Step 1 is complete
- The victim page currently uses Supabase client directly (line 93-101) which will work once RLS is fixed

### Step 7: Fix Seed Data
**File:** `rescuegrid/supabase/seed.sql`

- Line 44: Change `receiver_id` from victim_report UUID to `NULL`
- The DMA reply in a victim thread should not have a receiver_id

## DMA Messaging Tab Requirements (from PLAN.md Phase 8)

The DMA messaging panel should display:

### Channel List (Left Column)
- **Victim threads**: labeled `REPORT #KL-...` + situation badge
- **TF rooms**: labeled `TF · [name]`
- **Direct messages**: labeled volunteer name
- Each item: last message preview (40 chars), timestamp, unread count badge
- Flagged channels: red ⚑ icon

### Thread Panel (Right Column)
- Same bubble style as victim status screen
- Victim messages: left-aligned, `--bg3` bg, show phone number
- DMA messages: right-aligned, `--orange-dim` bg, `DMA · COMMAND` label
- Flagged messages: red left border + ⚑ icon
- Unread: bold sender until `read_at` set
- Message input + `SEND →` → POST `/api/dma/message`

### Realtime Requirements
- All channels live via Supabase Realtime
- `read_at` updated when DMA opens a thread
