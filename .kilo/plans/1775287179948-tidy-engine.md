# Phase 9 — Broadcast + Resource Management Plan

## Overview

This plan implements Phase 9 of RescueGrid: **DMA Emergency Broadcast** (Volunteer-Only) and **Full Resource Management with Allocation Tracking**.

### Key Decisions
- **Volunteer-Only Broadcast:** Broadcasts go only to volunteers (internal responder coordination). Victims continue to receive communications via their report thread, not mass broadcast.
- **Resource Allocation:** New `resource_allocation` table tracks who received what resources, with full history visible to DMA and volunteers.

### Architecture
- **Broadcast:** Uses existing `message` table with `receiver_id` per volunteer (no new table needed)
- **Resource Allocation:** New table linking `resource` → `assignment`/`task_force`/`volunteer` with quantity tracking
- **Real-Time:** Supabase Realtime on both `resource` and `resource_allocation` tables

---

## Architecture Decisions

### No Broadcast Table Required

The database schema already supports broadcasts via the `message` table:

| Column | Value for Broadcast |
|--------|---------------------|
| `sender_type` | `'dma'` |
| `sender_id` | DMA user UUID or `null` |
| `receiver_id` | Target volunteer UUID (one row per recipient) |
| `task_force_id` | `null` |
| `victim_report_id` | `null` |
| `content` | Broadcast message text |

**Target Resolution:**
- `ALL_VOLUNTEERS` → Query `volunteer` WHERE `status = 'active'`
- `SPECIFIC_TASK_FORCE` → Join `task_force_member` → `volunteer` WHERE `task_force_id = X`
- `EVERYONE` → Query all `volunteer` rows

### Real-Time Strategy

Volunteers subscribe to `message` table where `receiver_id = their_volunteer_id`. New broadcast messages appear instantly. This uses the existing `useRealtimeSubscription` hook from `@/lib/realtime.ts`.

### Web Push Strategy

Each target volunteer's `push_token` is fetched and `sendPush()` is called for each recipient. Since there's no batch endpoint in the web-push library, we send individual POST requests. For large numbers, this is acceptable as it's a DMA-triggered action, not high-frequency.

## Database Changes

### New Table: `resource_allocation`

**Migration:** `supabase/migrations/003_resource_allocation.sql`

```sql
-- Resource allocation tracking - who got what
CREATE TABLE resource_allocation (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  resource_id uuid NOT NULL REFERENCES resource(id),
  
  -- Who received it (one of these must be set, rest null)
  assignment_id uuid REFERENCES assignment(id),        -- allocated to an assignment
  task_force_id uuid REFERENCES task_force(id),        -- allocated to a task force
  volunteer_id uuid REFERENCES volunteer(id),          -- allocated to individual volunteer
  
  -- Allocation details
  quantity_allocated float NOT NULL,                    -- how much given
  quantity_consumed float DEFAULT 0,                    -- how much was used
  quantity_returned float DEFAULT 0,                    -- how much came back
  
  -- Status tracking
  status text DEFAULT 'allocated',                     -- allocated | in_use | consumed | returned | lost
  
  -- Metadata
  notes text,                                          -- e.g., "Delivered to sector 7 camp"
  allocated_by uuid,                                   -- DMA user who approved (optional)
  allocated_at timestamptz DEFAULT now(),
  updated_at timestamptz DEFAULT now()
);

-- Enable Realtime
ALTER PUBLICATION supabase_realtime ADD TABLE resource_allocation;

-- Indexes for common queries
CREATE INDEX idx_resource_allocation_resource ON resource_allocation(resource_id);
CREATE INDEX idx_resource_allocation_assignment ON resource_allocation(assignment_id);
CREATE INDEX idx_resource_allocation_task_force ON resource_allocation(task_force_id);
CREATE INDEX idx_resource_allocation_volunteer ON resource_allocation(volunteer_id);
```

### Schema Update: Add `push_token` to `victim_report` (Optional - Future)

**Note:** Storing victim push tokens for emergency broadcast was discussed but **excluded from Phase 9**. Victims remain anonymous and receive communications only via their report thread. This can be revisited in V2 with SMS integration for true emergency broadcasts.

#### 9.2.2 Broadcast Send API

**Route: `app/api/dma/broadcast/route.ts`** (POST)

**Purpose:** Send broadcast to all target volunteers via message rows + Web Push.

**Request Body:**
```typescript
{
  message: string; // required, max 500 chars
  target: 'all_volunteers' | 'specific_task_force' | 'everyone'; // required
  taskForceId?: string; // required if target = 'specific_task_force'
}
```

**Validation:**
- 400 if message empty or > 500 chars
- 400 if target invalid
- 400 if target = 'specific_task_force' and taskForceId missing

**Logic:**

1. **Resolve Target Volunteer IDs:**
   ```typescript
   let volunteerIds: string[] = [];
   let volunteersWithTokens: { id: string; push_token: string | null }[] = [];
   
   if (target === 'all_volunteers') {
     const { data } = await supabase
       .from('volunteer')
       .select('id, push_token')
       .eq('status', 'active');
   } else if (target === 'everyone') {
     const { data } = await supabase
       .from('volunteer')
       .select('id, push_token');
   } else if (target === 'specific_task_force') {
     const { data } = await supabase
       .from('task_force_member')
       .select('volunteer_id, volunteer:volunteer_id(id, push_token)')
       .eq('task_force_id', taskForceId);
     // Extract volunteer ids and push_tokens
   }
   ```

2. **Insert Message Rows (One per recipient):**
   ```typescript
   const messageRows = volunteerIds.map(volunteerId => ({
     content: message,
     sender_type: 'dma',
     sender_id: null, // or DMA user ID from session if available
     receiver_id: volunteerId,
     task_force_id: null,
     victim_report_id: null,
     is_flagged_for_dma: false,
   }));
   
   const { data: insertedMessages } = await supabase
     .from('message')
     .insert(messageRows)
     .select();
   ```

3. **Send Web Push Notifications:**
   ```typescript
   const pushTitle = '🚨 Emergency Broadcast';
   const pushBody = message.substring(0, 100); // Truncate to 100 chars
   
   const pushPromises = volunteersWithTokens
     .filter(v => v.push_token)
     .map(v => sendPush(v.push_token!, pushTitle, pushBody));
   
   // Fire and forget - don't block response on push
   Promise.allSettled(pushPromises).catch(console.error);
   ```

4. **Return Response:**
   ```typescript
   return NextResponse.json({ 
     sent: volunteerIds.length,
     message: `Broadcast sent to ${volunteerIds.length} volunteers`
   }, { status: 201 });
   ```

**Error Handling:**
- 400: Validation errors
- 500: Database or push errors (return error but log details)

#### 9.4.4 Allocate Resource (NEW)

**Route:** `app/api/dma/resource/allocate/route.ts` (POST)

**Purpose:** Create allocation from resource to assignment/TF/volunteer.

**Request Body:**
```typescript
{
  resource_id: string;           // required
  quantity: number;              // required, must be <= available
  assignment_id?: string;        // one of these three
  task_force_id?: string;
  volunteer_id?: string;
  notes?: string;
}
```

**Validation:**
- 400: No target specified (assignment/TF/volunteer)
- 400: Quantity > available
- 400: Quantity <= 0
- 404: Resource not found

**Logic:**
```typescript
// 1. Check available quantity
const { data: resource } = await supabase
  .from('resource')
  .select('quantity')
  .eq('id', resource_id)
  .single();

const { data: allocations } = await supabase
  .from('resource_allocation')
  .select('quantity_allocated')
  .eq('resource_id', resource_id)
  .in('status', ['allocated', 'in_use']);

const allocated = allocations?.reduce((sum, a) => sum + a.quantity_allocated, 0) || 0;
const available = resource.quantity - allocated;

if (quantity > available) {
  return NextResponse.json({ error: 'Insufficient stock' }, { status: 400 });
}

// 2. Create allocation
const { data: allocation } = await supabase
  .from('resource_allocation')
  .insert({
    resource_id,
    assignment_id,
    task_force_id,
    volunteer_id,
    quantity_allocated: quantity,
    status: 'allocated',
    notes,
  })
  .select()
  .single();

// 3. Update resource updated_at (to trigger realtime)
await supabase
  .from('resource')
  .update({ updated_at: new Date().toISOString() })
  .eq('id', resource_id);
```

**Response:** 201 with created allocation

- Only volunteer who `receiver_id` matches can update
- OR volunteers in the task_force can update shared allocations
- DMA can always override (separate DMA API)

## Edge Cases & Handling

### Broadcast

| Edge Case | Handling |
|-----------|----------|
| Empty message | Validation error, disable broadcast button |
| No volunteers match target | Show "0 recipients" warning, disable broadcast |
| All push tokens invalid | Log errors, broadcast still succeeds (messages created) |
| Large recipient count (>100) | Batch insert in chunks of 100, show progress |
| Network failure during broadcast | Retry once, show partial success if needed |
| Volunteer has no push token | Skip push, message still created in DB |

### Resource Allocation

| Edge Case | Handling |
|-----------|----------|
| Insufficient stock (requested > available) | API validates, returns 400 with "Insufficient stock" |
| Allocation to invalid assignment/TF | API validates FK exists, returns 404 |
| Negative allocation quantity | API rejects, min = 1 |
| Resource deleted while allocated | Soft delete? Or prevent deletion if active allocations exist |
| Partial consumption | Support `quantity_consumed` < `quantity_allocated` |
| Over consumption | Validate `quantity_consumed` <= `quantity_allocated` |
| Volunteer updates TF allocation | Allow (team coordination), but log who updated |
| DMA overrides volunteer status | DMA PATCH always allowed, volunteer PATCH restricted |
| Zero available resources | Disable "ALLOCATE" button, show "OUT OF STOCK" badge |

### Resource Management

| Edge Case | Handling |
|-----------|----------|
| Resource quantity negative | Clamp to 0 in API |
| Duplicate resource name | Allow duplicates (different types/locations possible) |
| Editing quantity below allocated | Warn "X units already allocated", allow but flag |
| Realtime conflict (two DMAs editing) | Last-write-wins with merge conflict UI |

### Volunteer Resource View

| Edge Case | Handling |
|-----------|----------|
| Volunteer not in any TF | Hide "Task Force Shared" section |
| No resources allocated | Show empty state "NO RESOURCES ASSIGNED" |
| Allocation marked consumed twice | Idempotent - same result if already consumed |

## RLS (Row Level Security) Policies

For new table `resource_allocation`:

| Role | SELECT | INSERT | UPDATE | DELETE |
|------|--------|--------|--------|--------|
| anon | ❌ | ❌ | ❌ | ❌ |
| volunteer (authenticated) | ✅ WHERE volunteer_id = auth.uid() OR task_force_id in (user's TFs) | ❌ | ✅ WHERE volunteer_id = auth.uid() | ❌ |
| DMA (service role) | ✅ | ✅ | ✅ | ✅ |

**Note:** Volunteers can only update status/consumed/returned fields, not reallocate.

## Security Considerations

1. **Authentication:** All broadcast/resource routes require DMA session
2. **Rate Limiting:** Consider rate limiting broadcast endpoint (1 per 10 seconds)
3. **Input Sanitization:** Message content sanitized before insert
4. **No Volunteer Broadcast:** Volunteers cannot send broadcasts (enforced by RLS + API auth)

---

## Performance Considerations

1. **Broadcast to Large Numbers:**
   - Use `supabase.from().insert()` with array (supports bulk insert)
   - Fire push notifications asynchronously (don't await)
   - Return immediately after database insert

2. **Resource Realtime:**
   - Single subscription to `resource` table
   - Update local state optimistically

3. **Recipient Count Preview:**
   - Cache counts for 5 seconds to prevent rapid re-querying
   - Debounce target selection changes
